# 논문 리뷰·정리 웹 서비스 기획서

> 작성일: 2026-07-03 | 버전: v4.0 | 작성자: PM

> **v3.0 변경 요지:** AI의 역할을 "자동 생성자"에서 "요청 시 돕는 보조"로 격하. 사용자의 직접 작성을 제품의 중심에 둔다. AI 없이도 가치를 주는 제품으로 설계하고 AI는 선택적 레이어로 분리.

> **v3.1 수정 요지:** 레이어 분류 오류 정정, 인증 스택 교체, 사용자 질문 기록 코어 분리, 스토리보드 재정렬, 용어 힌트 한계 명시.

> **v4.0 변경 요지:** 실 구현·운영 경험을 반영한 방향 전환.
> 1. **AI 내용 요약을 보조 레이어에서도 제외**한다(구 FR-13/14 요약 초안, FR-15 Q&A 폐기). 대신 **요약 이외의 모든 반복 작업(등록·추출·메타데이터·분류·내보내기)을 자동화**하는 데 집중한다.
> 2. 연구 주제에 맞는 **논문 탐색(검색·추천)은 아직 도울 수 없음을 범위로 명시**하고, 대신 확보한 논문에서 **키워드·관점·한계·비판점을 빠르게 찾아내는 템플릿과 규칙 기반 자동화**를 제공한다.
> 3. 범용 5문항 템플릿 하나에서 **작업 목표별 목적 특화 템플릿 세트**로 확장한다. 템플릿의 목적을 좁히고 종류를 다양화한다.
> 4. 기술 스택·데이터 모델·화면 구조를 실제 배포 상태(GitHub Pages + Render + Supabase Auth, 분리 테이블 저장소)로 갱신한다.

## 1. 서비스 개요

**서비스명:** PaperLens

**한 줄 정의:** 논문을 직접 읽는 사람의 "읽기 이외의 모든 작업"을 자동화하고, 읽기 자체는 작업 목적에 맞는 템플릿으로 구조화하는 리뷰 노트 워크스페이스.

**v4.0 핵심 컨셉:** 내용 이해와 요약은 사용자의 몫으로 남긴다 — AI가 대신 읽지 않는다. 그 대신 등록, 텍스트 추출, 메타데이터, 분류, 진행 관리, 내보내기 등 읽기 주변부의 반복 작업은 도구가 철저히 자동화한다. 그리고 사용자가 논문에서 무엇을 봐야 하는지(키워드·관점·한계·비판점)를 목적별 템플릿과 규칙 기반 안내로 좁혀 준다.

**v3.x에서의 연속성과 단절:**

| 항목 | v3.1 | v4.0 |
|------|------|------|
| 사용자 주도 작성 | 중심 원칙 | 유지·강화 |
| AI 용어 설명 (요청 시) | 보조 레이어 | 유지 (유일한 AI 기능) |
| AI 요약 초안 (FR-13/14) | 보조 레이어, Phase 2 | **폐기** — 제품 목적과 상충 |
| 논문 기반 Q&A (FR-15) | 보조 레이어, Phase 2 | **폐기** |
| 다중 논문 비교 (FR-17) | AI 보조, P3 | AI 없이 "라이브러리 취합"으로 축소 재정의 |
| 인용 네트워크 (FR-18) | 외부 API, P3 | **보류** (탐색 기능은 범위 밖) |
| 수동 요약 템플릿 | 범용 5문항 1종 | 목적별 템플릿 5종으로 확장 |
| 자동화 | 등록·파싱·메타 추출 | 요약 외 전 작업으로 확대 + 시그널 문장 스캐너 신설 |

## 2. 타겟 고객층

| 구분 | 세부 내용 |
|------|-----------|
| 1차 타겟 | 국내외 대학원생(석·박사), 연구자 |
| 2차 타겟 | 논문 읽어야 하는 스타트업 개발자·PM, 기업 R&D 팀 |
| 3차 타겟 | 의학·법학·특허 등 전문직 종사자 |
| 연령대 | 22~45세 |
| 사용 환경 | PC 중심(논문 작업), 모바일 보조 |
| 기술 친숙도 | 중~상 (AI 도구 사용 경험 有) |

**v4.0 보충 — 타겟의 "작업 목표" 세분화:** 같은 사용자라도 논문을 읽는 목적이 매번 다르다. v4.0은 타겟을 인구통계가 아니라 **작업 목표**로 다시 자른다. 이 축이 목적별 템플릿(§8-3)의 근거다.

| 작업 목표 | 대표 상황 | 대응 템플릿 |
|-----------|-----------|--------------|
| 정독 여부 판단 | 지도교수가 준 논문 더미 훑기 | T1 빠른 훑어보기 |
| 선행연구 정리 | 학위논문 서론·Related Work 집필 | T2 선행연구 정리 |
| 방법 이식 | 실험 설계·분석 방법 벤치마킹 | T3 방법론 벤치마킹 |
| 비판적 검토 | 세미나 발제, 리뷰어 역할, 반론 구성 | T4 비판적 검토 |
| 수치 비교 | 내 결과와 성능·효과크기 비교 | T5 결과 비교·수치 수집 |

## 3. 최근 트렌드 분석

### 3-1. 시장 트렌드

- "AI 자동 생성물에 대한 신뢰 피로(trust fatigue)" 지속 — 사용자가 직접 검증·작성하는 도구 선호 증가 (v3.0 판단 유지, v4.0에서 더 밀고 나감)
- 능동적 학습(active recall) 도구가 수동적 요약 소비보다 학습 효과 측면에서 재평가받는 흐름
- AI 요약 도구가 범람하면서, 역으로 "요약은 내가 하되 잡무를 없애 주는" 도구 포지션이 비어 있음

### 3-2. 경쟁 서비스 현황

| 서비스 | 특징 | 한계 |
|--------|------|------|
| **Elicit** | 1,000편 자동 스크리닝 | 영어 중심, 한국어 약함 |
| **Consensus** | 2억 편 DB, LLM 기반 답변 | 한국어 미지원 |
| **Semantic Scholar** | 2억 편+ 무료 검색 | 요약·정리 기능 제한적 |
| **ResearchRabbit** | 인용 네트워크 시각화 | 요약·리뷰 기능 부재 |
| **Scholarcy** | PDF 구조화 요약 | 유료, 한국어 지원 부족 |

> **v4.0 차별화 포인트:** 경쟁사는 "논문을 찾아 주거나(탐색)" "내용을 만들어 주는(요약)" 도구다. PaperLens는 그 둘을 하지 않는다고 선언하는 대신, (1) 요약 이외의 작업을 전부 자동화하고 (2) 읽기 목적별로 무엇을 건져야 하는지 구조화해 주는 유일한 포지션을 갖는다. 한국어 논문(KCI 등 CrossRef 미등재 포함)의 추출·메타데이터 파이프라인이 실동작한다는 점도 실질 차별점이다.

## 4. 기획 방향 (To-Be)

> **차별점: 한국어 최적화 + 요약 제외 전 작업 자동화 + 목적별 읽기 템플릿**

### 4-1. 제품 3원칙 (v4.0 개정)

v3.0의 "AI 보조 3원칙"(사용자가 주체 / 요청 시에만 작동 / 수락을 거쳐 확정)은 유지하되, 상위에 제품 원칙을 새로 둔다.

1. **요약은 사용자, 잡무는 도구** — 내용 이해·요약·비판은 사용자가 한다. 등록·추출·메타데이터·분류·저장·내보내기는 도구가 자동으로 한다.
2. **탐색은 안 하고, 발굴은 돕는다** — 논문을 찾아 주지 않는다. 대신 확보한 논문 안에서 키워드·관점·한계·비판점을 빨리 찾도록 템플릿과 규칙 기반 안내를 제공한다.
3. **템플릿은 목적별로 좁게** — 만능 템플릿 하나 대신, 작업 목표마다 질문·권장 라벨·완료 기준·출력 형식이 다른 전용 템플릿을 제공한다.

### 4-2. 자동화의 경계 (v4.0 핵심)

v3.x의 "코어(AI 비의존) / 보조(AI 선택)" 레이어 구분은 유지한다. 단, 보조 레이어에 들어갈 수 있는 AI 기능의 심사 기준을 신설한다.

> **심사 기준: "내용을 대신 이해하는가?"** — 그렇다면 도입하지 않는다.
> - AI 요약 초안, 논문 Q&A: 내용을 대신 이해함 → **제외**
> - AI 용어 설명: 배경지식 보조이지 본문 요약이 아님 → 유지 (요청 시, 수락 후 확정)
> - 시그널 문장 스캐너: 규칙 기반으로 "위치"만 안내, 이해는 사용자 몫 → 코어로 도입

자동화는 **규칙 기반·결정적(deterministic)** 을 우선한다. 자동 결과는 항상 근거와 함께 표시하고, 사용자 수정이 자동 결과보다 우선한다(`user_edited` 우선 원칙).

### 4-3. 핵심 기능 및 구현 상태

v3.1의 "구현 가능성" 열을 v4.0에서는 "구현 상태"로 바꾼다. 상당수가 이미 배포·운영 중이다.

## 5. Pain Point

1. **시간 비용** - 논문 1편 읽고 이해하는 데 평균 2~4시간 소요
2. **영어 장벽** - 비영어권 대학원생의 전문 영어 독해 부담
3. **맥락 파악 어려움** - 논문 하나만으로는 연구 흐름 파악 한계
4. **리뷰 작성 막막함** - 어떻게 비판·요약해야 할지 모르는 초보 연구자
5. **정보 파편화** - 읽은 논문, 메모, 인용이 여러 도구에 분산됨
6. **재현 불가** - "저번에 읽었는데 어디 저장했더라?" 문제
7. **AI 요약 불신** - AI 요약을 그대로 믿기 어렵고 검증 과정이 분리되어 비효율

> **v4.0 해결 구조 재배치:** v3.1은 #1·#2(독해 부담)를 "AI 보조(Phase 2)가 본격적으로 다룬다"고 했다. AI 요약을 폐기한 v4.0에서는 이렇게 다시 맡긴다.
>
> | Pain | v3.1의 답 | v4.0의 답 |
> |------|-----------|-----------|
> | #1 시간 비용 | AI 요약 초안 | **읽기 외 시간 제거**(자동화 §4-3) + **봐야 할 곳 안내**(시그널 스캐너 §8-5, 목적별 템플릿 §8-3) |
> | #2 영어 장벽 | AI 설명·초안 | 용어 사전 + AI 용어 설명(유지). 본문 번역은 별도 번역 보기 패널로 후속 검토(브라우저 자동 번역은 하이라이트 DOM과 충돌해 비지원) |
> | #3 맥락 파악 | 비교·네트워크(P3) | 라이브러리 취합(§8-6) — 탐색 없이 "내가 읽은 것"의 취합으로 축소 |
> | #4 작성 막막함 | 수동 템플릿+AI 초안 | **목적별 템플릿** — 무엇을 물어야 할지 자체를 템플릿이 제공. AI 초안 없이 질문 설계로 해결 |
> | #5·#6 파편화·재현 | 지식베이스 | 유지 (사용자별 저장, 취합 뷰로 강화) |
> | #7 AI 요약 불신 | 수락 절차 | **원천 해소** — 요약 AI가 없으므로 불신 대상 자체가 없음 |

## 7. 페르소나

### 페르소나 A — "논문 폭탄 맞은 석사생" 김지수 (26세)

- **직업/상황:** 생명과학 석사 1년차, 지도교수에게 주 3편 논문 리뷰 과제
- **목표:** 내가 이해한 내용으로 리뷰 노트를 직접 작성하되, 잡무 없이 읽기에만 집중하기
- **Pain Point:**
  - 논문마다 서지정보 정리·파일 관리에 시간을 뺏김
  - 리뷰를 쓰라는데 "한계·비판점"을 어디서 찾아야 할지 모름
  - 모르는 전문 용어가 너무 많음
- **v4.0 기대 가치:** "등록하면 제목·저자·태그가 알아서 채워지고, **비판적 검토 템플릿이 '저자가 말하지 않은 한계는?' 같은 질문을 던져 주니** 뭘 찾으며 읽어야 할지 안다. 한계 시그널 밑줄을 따라가면 빠르다."

### 페르소나 B — "학위논문 서론 쓰는 박사수료생" 박서연 (31세)

- **직업/상황:** 교육학 박사수료, 학위논문 선행연구 절 집필 중. 읽어 둔 논문 40여 편
- **목표:** 논문별로 "내 연구와의 연결·차별점·인용 문장"을 정리하고, 나중에 한 번에 회수하기
- **Pain Point:**
  - 읽을 때는 정리한 것 같은데 집필 시점에 어느 논문 어느 문장인지 재현 불가
  - 인용 목적(전제/비교/반론)별로 문장을 다시 분류하는 데 하루가 감
- **v4.0 기대 가치:** "선행연구 정리 템플릿으로 읽고, 하이라이트에 인용 목적만 붙여 두면, **라이브러리 취합에서 '반론' 문장만 논문 출처와 함께 한 번에 뽑힌다.**"

> v3.1의 페르소나 B(스타트업 PM)는 2차 타겟으로 유지하되, v4.0의 대표 페르소나는 제품 방향(집필 준비·비판적 읽기)과 직결되는 위 두 명으로 교체한다.

### 8-2. 핵심 용어 사전 — 사용자 주도 설계 (v3.0 승계)

v3.1 §8-2의 설계(사용자가 모르는 단어를 직접 지목, 규칙 기반 옅은 밑줄 힌트, AI 설명은 요청 시)를 그대로 유지한다. 힌트 정확도의 한계(순한글 전문용어 미탐지)도 동일하게 인정하며, v4.0의 시그널 스캐너와 표시 체계를 통일한다(비침습적 안내 → 클릭 시 확정이라는 동일 패턴).

### 8-3. 목적별 요약 템플릿 (v4.0 핵심 변경 — 구 8-3 확장)

**기존: 범용 5문항 1종 → 변경: 작업 목표별 5종.** 템플릿 = (목적, 질문 세트, 권장 라벨, 완료 기준, 출력 형식)의 묶음. 노트 생성 시 선택하고 이후 변경 가능하다.

**T1. 빠른 훑어보기** (기존 5문항 승계 — 하위 호환)
- 목적: 30분 내 뼈대 파악, 정독 여부 판단
- 질문: ① 무엇을 해결하려 하는가 ② 어떤 방법을 사용했는가 ③ 결과는 무엇인가 ④ 한계는 무엇인가 ⑤ 내가 이해한 핵심은 무엇인가
- 권장 라벨: 주장, 결과 / 완료 기준: 5문항 작성

**T2. 선행연구 정리** (Related Work 집필 준비)
- 목적: 내 논문의 서론/선행연구 절에 인용할 재료 수집
- 질문: ① 이 논문이 내 연구와 어떻게 연결되나 ② 내 논문에서 한 문장으로 소개한다면 ③ 내 연구와의 차별점은 ④ 어떤 맥락(전제·비교·반론)에서 인용할 것인가
- 권장 라벨: 주장, 근거 / 권장 인용 목적: 전제 인용, 관련 연구
- 완료 기준: 인용 후보 1개 이상 + "한 문장 소개" 작성 / 출력: 인용 후보 보드 우선

**T3. 방법론 벤치마킹**
- 목적: 연구 설계·실험·분석 방법을 내 연구에 이식/비교
- 질문: ① 연구 설계는(대상·표본·조건) ② 핵심 절차·도구·지표는 ③ 내 상황에 적용 시 바꿔야 할 것은 ④ 이 방법의 전제 조건과 취약점은
- 권장 라벨: 방법론 / 권장 인용 목적: 방법 참고
- 완료 기준: 방법론 하이라이트 3개 이상 + 적용 계획 작성

**T4. 비판적 검토** (키워드·관점·한계·비판점 발굴 — §8-5와 직결)
- 목적: 세미나 발제·리뷰·반론 구성. 논문을 뜯어보는 읽기
- 질문: ① 저자의 관점/전제는 무엇인가 ② 저자가 인정한 한계는 ③ 저자가 말하지 않은 한계는 ④ 주장-근거 사슬에서 가장 약한 고리는 ⑤ 결과가 성립하지 않는 조건은
- 권장 라벨: 한계/비판, 질문/후속 확인, 근거 / 권장 인용 목적: 반론, 한계 언급
- 완료 기준: `한계/비판` 하이라이트 2개 이상 + "말하지 않은 한계" 작성
- 시그널 스캐너를 기본 활성화하는 유일한 템플릿

**T5. 결과 비교·수치 수집**
- 목적: 성능표·효과크기 등 내 결과와 비교할 수치 수집
- 질문: ① 비교 대상 지표와 값은 ② 실험 조건이 내 것과 같은가 다른가 ③ 직접 비교 가능한가, 보정이 필요한가
- 권장 라벨: 결과 / 권장 인용 목적: 결과 비교
- 완료 기준: `결과 비교` 인용 후보 1개 이상

> 각 질문 카드에는 관련 라벨의 하이라이트만 모아 보는 필터가 붙는다(예: T4 ②번 카드에서 `한계/비판` 하이라이트 목록). "무엇을 물어야 할지"는 템플릿이, "본문 어디를 볼지"는 라벨·스캐너가, "답"은 사용자가 채운다.

### 8-4. 라벨 → 인용 목적 기본값 (v4.0 신규 자동화)

하이라이트에 의미 라벨을 붙이는 순간 인용 목적 기본값을 제안해 클릭 수를 줄인다. 제안은 언제나 덮어쓸 수 있으며 자동 확정하지 않는다.

| 의미 라벨 | 제안 기본 인용 목적 |
|-----------|---------------------|
| 주장 | 전제 인용 |
| 방법론 | 방법 참고 |
| 결과 | 결과 비교 |
| 한계/비판 | 한계 언급 |
| 근거 | 전제 인용 |
| 질문/후속 확인 | (제안 없음 — 사용자 판단) |

### 8-5. 시그널 문장 스캐너 (v4.0 신규 — 규칙 기반, AI 아님)

본문에서 키워드·관점·한계·비판점의 **후보 문장 위치**를 규칙 기반으로 표시하는 "형광펜 가이드". AI 요약이 아니며, 이해·판단은 사용자 몫이다.

| 대상 | 규칙(예시) | 비고 |
|------|------------|------|
| 키워드 후보 | 기존 용어 힌트 패턴 확장: 빈도 × 초록/키워드 섹션 등장 × 대문자 약어. 논문에 키워드 섹션이 있으면 파싱해 최우선 제시 | 용어 사전과 연동 |
| 관점 시그널 | "we argue", "this paper proposes", "…관점에서", "…를 전제로", "우리는 …라고 본다" 등 입장 표명 패턴 | |
| 한계 시그널 | "limitation", "however", "future work", "제한점", "한계", "추후 연구", "…하지 못했다" 등. Discussion/Conclusion 섹션 가중 | |
| 비판점 후보 | 직접 탐지 불가. 표본/데이터 규모 언급, 가정 표명("assume", "가정"), 통계 유의성 경계 문장을 "비판적으로 볼 후보"로 표시 | 비판 작성은 사용자 |

**동작 원칙:**
- 원문 패널에 점선 밑줄 등 **비침습적 표시**. 클릭 한 번으로 해당 라벨 하이라이트로 승격
- 스캐너 결과는 저장하지 않는 휘발성 안내(승격된 하이라이트만 저장) — 오탐이 노트를 오염시키지 않음
- 프론트엔드 순수 함수 모듈(`frontend/src/lib/`)로 구현해 단위 테스트 필수
- 용어 힌트(§8-2)와 동일한 "안내→사용자 확정" 패턴이므로 UI 문법을 통일

### 8-6. 라이브러리 취합 (v4.0 신규 — 구 FR-17의 축소 재정의)

논문을 찾아 주지는 못해도, **이미 읽은 논문에서 무엇을 건졌는지**는 한눈에 회수한다.

- 내 라이브러리 전체에서 라벨/인용 목적별 취합 뷰 (예: 모든 논문의 `한계/비판` 하이라이트, 모든 `반론` 인용 후보)
- 취합 결과의 Markdown 내보내기(논문별 출처 표기 포함) — 선행연구 절 초안의 재료
- AI 비교 분석이 아니라 사용자가 분류해 둔 것의 집계이므로 코어 레이어(AI 비의존)

## 11. 기능 명세 (Functional Specification)

> FS-01(등록·파싱), FS-02(하이라이트), FS-03(용어 사전), FS-04(수동 작성·자동 저장)는 v3.1 명세가 구현으로 검증되었으므로 유지한다. v4.0 신규 명세만 추가한다.

### FS-06: 목적별 템플릿 [코어] 🆕

- **입력:** 노트 생성 시(또는 이후) 사용자의 템플릿 선택 (기본값 T1)
- **처리:** 템플릿 정의(질문 세트·권장 라벨·완료 기준·출력 구성)를 선언적 데이터로 로드 → 요약 템플릿 영역·진행 로드맵·내보내기 기본값 재구성 → 답변은 노트 JSON에 `templateId`와 함께 저장
- **출력:** 템플릿별 질문 카드(관련 라벨 필터 포함), 템플릿 기준 진행률
- **마이그레이션:** 기존 노트는 질문 키가 동일한 T1으로 무손실 이전. 템플릿 변경 시 기존 답변은 보존하고 질문 매핑이 없는 답변은 "이전 템플릿 답변"으로 접어 보관
- **예외:** 템플릿 정의 로드 실패 시 T1로 폴백

### FS-07: 시그널 문장 스캐너 [코어] 🆕

- **입력:** 추출된 원문 텍스트(+섹션 정보), 활성 템플릿
- **처리:** 규칙 사전(한/영 패턴, 섹션 가중치) 매칭 → 유형별(키워드/관점/한계/비판 후보) 후보 목록·오프셋 산출 → 원문 패널에 비침습 표시
- **출력:** 후보 밑줄 + 유형 툴팁 + "하이라이트로 추가" 액션(해당 라벨로 승격)
- **원칙:** 결과 비저장(승격분만 저장), T4에서 기본 켬·그 외 수동 켬, 규칙은 설정으로 유형별 on/off
- **예외:** 원문 품질 `낮음`/`추출 실패`이면 스캐너 비활성 + 안내

### FS-08: 라벨→인용 목적 기본값 [코어] 🆕

- **입력:** 하이라이트 의미 라벨 지정 이벤트
- **처리:** §8-4 매핑표로 기본값 제안(미확정 상태 표시) → 사용자가 그대로 두면 보드 분류에 사용, 변경하면 변경값 사용
- **출력:** 인용 후보 보드 자동 분류 (제안 출처 표시)

### FS-09: 라이브러리 취합 [코어] 🆕

- **입력:** 사용자 라이브러리의 전체 노트(목록 API는 본문 제외 경량 조회)
- **처리:** 라벨/인용 목적/태그 필터 → 논문별 그룹핑 → 출처(논문 제목·저자·연도) 표기
- **출력:** 취합 뷰 + Markdown 내보내기
- **예외:** 노트 수가 많을 때 페이지네이션·지연 로드

## 13. 데이터 모델 (구 ERD 갱신)

실 구현은 접근 패턴에 따라 분리 테이블을 사용한다(v3.1 ERD의 정규화 구상을 대체).

**저장 구조 (SQLite 기본, `DATABASE_URL` 설정 시 PostgreSQL):**

| 테이블 | 내용 |
|--------|------|
| `paper_metadata` | 제목, 저자, DOI, sourceKey, 메타데이터 출처·신뢰도·경고, PDF 파일명 등 목록/검색용 경량 정보 |
| `paper_texts` | 원문 텍스트 + `extractionQuality`. 목록 조회에서 읽지 않고 단일 노트에서 지연 로드 |
| `review_notes` | 리뷰 노트 JSON — 하이라이트(라벨·오프셋), 인용 목적, 태그, 질문, 용어, 템플릿 답변 |
| `paper_files` | PDF 파일명·바이너리 |

**v4.0 노트 JSON 확장 (서버 스키마 변경 불필요):**

- `templateId`: 선택된 템플릿 (기본 `t1_quick`) — 기존 노트는 T1으로 무손실 마이그레이션
- 템플릿별 답변 맵, 하이라이트별 인용 목적(제안/확정 구분)
- 스캐너 결과는 저장하지 않음(휘발성). 승격된 하이라이트만 기존 구조로 저장
- 템플릿 정의 자체는 DB가 아닌 프론트엔드 선언 모듈(`frontend/src/lib/templates.ts`)에 둔다

> 사용자 식별은 Supabase Auth JWT로 하며, 노트·PDF 접근은 검증된 `user_id`로 제한한다. 미설정 로컬 환경은 `local` 단일 사용자로 동작.

## 16. 개발 우선순위 (Roadmap)

> v3.1의 Phase 1(코어 MVP)은 완료·운영 중이다. v3.1 Phase 2(AI 보조 도입) 중 용어 설명만 반영됐고 요약 초안·Q&A는 폐기한다. v4.0 로드맵은 그 위에서 시작한다.

| 단계 | 내용 | 근거 |
|------|------|------|
| **M1** | 템플릿 데이터 모델(`templateId`) + 템플릿 선택 UI + **T1/T4** 출시 | 원칙 3의 골격. T4가 관점·한계·비판점 발굴(원칙 2)의 수동 경로를 바로 연다 |
| **M2** | 라벨→인용 목적 기본값(FR-23), 템플릿별 완료 기준·내보내기(FR-20/21) | 기존 기능 위 저비용 자동화(원칙 1) |
| **M3** | T2/T3/T5 추가 + 질문 카드 ↔ 라벨 필터 연동(FR-22) | 템플릿 다양화 완성 |
| **M4** | 시그널 스캐너 — **키워드·한계 시그널부터**(FR-24 부분) | 규칙 기반이라 검증 용이. 관점·비판 시그널은 승격률 데이터 확인 후 확장 |
| **M5** | 라이브러리 취합 뷰·취합 내보내기(FR-25) | 다논문 가치의 최소 형태 |

각 단계는 기존 CI 검증(백엔드 ruff+pytest, 프론트 eslint+vitest+build)을 통과해야 하며, 스캐너·진행률 규칙은 순수 함수 + 단위 테스트를 필수로 한다.

**성공 지표:**

- 등록→읽기 시작까지 사용자 개입 횟수 (자동화 지표, 감소 목표)
- T1 이외 목적 템플릿 선택률 (목적 특화가 실제로 쓰이는가)
- T4 노트의 `한계/비판`·`질문/후속 확인` 하이라이트 수 (비판점 발굴 지표)
- 스캐너 후보 → 하이라이트 승격률 (정밀도 대리 지표. 낮으면 해당 규칙 축소)
- 노트/취합 내보내기 실행률 (산출물로 이어지는가)

## 17. 수익 모델

AI 요약을 폐기했으므로 v3.1의 "AI 보조 = 유료 차별점" 구조를 바꾼다. 유료 차별점은 **목적별 템플릿 전체 + 발굴·취합 도구**다.

| 플랜 | 가격 | 내용 |
|------|------|------|
| Free | 무료 | 월 5편 리뷰 노트, 코어 작성 기능 전체, T1 템플릿 |
| Pro | 9,900원/월 | 무제한 리뷰 노트, **목적별 템플릿 전체(T2~T5)**, **시그널 스캐너**, **라이브러리 취합·내보내기**, AI 용어 설명 |
| Team | 39,900원/월 | Pro + 협업·공유·관리자 기능 (기능 자체가 보류 상태이므로 도입 시점 재검토) |

> AI 비용 노출이 용어 설명(짧은 호출)로 줄어 원가 리스크가 v3.1보다 더 작아진다. 유료 가치가 "AI가 해 준다"가 아니라 "내 작업 방식에 맞는 도구"로 옮겨가므로, 신뢰 피로 트렌드(§3-1)와도 정합적이다.
