# PaperLens 제품 기획서 v4.0

- 문서 버전: 4.0
- 작성일: 2026-07-03
- 이전 버전: v3.1 (로컬 전용 문서)
- 상태: 방향 전환 반영 — v3.1의 "AI 보조 요약 중심 리뷰 도구"에서 "요약을 제외한 읽기 작업 자동화 + 목적별 템플릿 도구"로 재정의

---

## 1. 버전업 배경

v3.1까지의 기획은 "AI가 논문 내용을 요약해 주는 것"을 장기 로드맵의 중심에 두었다(요약 템플릿 AI 초안, 논문 기반 Q&A 등). 그러나 프로젝트를 진행하며 다음이 확인되었다.

1. **AI 내용 요약은 제품의 목적과 어긋난다.** PaperLens의 가치는 사용자가 논문을 *직접 읽으며* 이해를 구조화하는 데 있다. AI가 내용을 대신 요약하면 이 가치가 훼손되고, 요약 품질 검증 부담만 늘어난다. 따라서 **AI 내용 요약은 제품 범위에서 제외**한다.
2. **논문 탐색(검색/추천)은 아직 도울 수 없다.** "연구 주제에 맞는 논문을 찾아주는 기능"은 현재 기술 스택과 데이터로는 신뢰할 수 있는 수준으로 제공할 수 없다. 대신, 사용자가 **이미 확보한 논문에서 키워드·관점·한계·비판점을 빠르게 찾아내는 것**을 돕는 데 집중한다.
3. **범용 요약 템플릿 하나로는 부족하다.** 사용자가 논문을 읽는 목적(선행연구 정리, 방법론 벤치마킹, 비판적 검토, 논문 집필 준비 등)에 따라 필요한 질문과 결과물이 다르다. 템플릿의 목적을 좁히고, **작업 목표별로 다양한 템플릿**을 제공한다.

이에 따라 제품의 방향을 아래 세 축으로 재정의한다.

| 축 | v3.1 | v4.0 |
| --- | --- | --- |
| AI 역할 | 요약 초안, Q&A까지 확장 예정 | 내용 요약 제외. 요약 이외의 반복 작업(등록·추출·분류·내보내기) 자동화 |
| 논문 탐색 | (장기) 추천/비교 구상 | 범위 제외. 대신 확보한 논문의 키워드/관점/한계/비판점 발굴을 템플릿·자동화로 지원 |
| 템플릿 | 범용 5문항 요약 템플릿 | 작업 목표별 목적 특화 템플릿 세트 |

## 2. 제품 정의

**PaperLens는 논문을 직접 읽는 사람의 "읽기 이외의 모든 작업"을 자동화하고, 읽기 자체는 목적에 맞는 템플릿으로 구조화하는 웹 워크스페이스다.**

- 사용자: 논문을 읽고 정리해야 하는 대학원생·연구자·학부 고학년
- 핵심 약속:
  - 내용 이해와 요약은 사용자의 몫으로 남긴다. (AI가 대신 읽지 않는다)
  - 그 외의 모든 것 — 등록, 텍스트 추출, 메타데이터, 분류, 진행 관리, 내보내기 — 은 도구가 처리한다.
  - 사용자가 무엇을 봐야 하는지(키워드·관점·한계·비판점)를 템플릿과 규칙 기반 자동화로 안내한다.

### 범위 제외 (Non-goals)

- **AI 본문 요약 / 요약 초안 생성**: 제공하지 않는다. (용어 설명 초안은 "내용 요약"이 아니라 배경지식 보조이므로 선택 기능으로 유지)
- **논문 검색/추천**: 연구 주제에 맞는 논문을 찾아 주지 않는다. 외부 검색 포털 링크(KCI, RISS, KISS, Google Scholar) 제공까지만 한다.
- **AI 논문 기반 Q&A**: 제외한다.
- 공유/협업: 단일 사용자 도구로 유지하고 협업은 후순위로 남긴다.

## 3. 현재 구현 현황 (v4.0 기준선)

v4.0은 백지 기획이 아니라 이미 배포·운영 중인 기능 위에서의 방향 전환이다. 현재 상태:

### 3.1 자동화가 이미 구현된 영역

- **논문 등록**: PDF 업로드, PDF 원문 URL 다운로드(`/api/papers/extract-url`), DOI 메타데이터 등록, 샘플 PDF 원클릭 등록(중복 방지 sourceKey 포함)
- **텍스트 추출**: PyMuPDF 기반 추출 + 1단/2단/혼합 레이아웃 판정 reflow, 글자 벌어짐 보정, 헤더/푸터/arXiv 스탬프 노이즈 제거, 추출 실패 시 raw fallback
- **추출 품질 자동 진단**: `extractionQuality` 점수(0-100)·상태·근거를 자동 산출해 업로드 노티와 원문 패널에 표시. 품질이 낮으면 텍스트 편집/직접 입력으로 사용자 보정
- **메타데이터 자동 보완**: PDF 내장 메타데이터 → DOI 탐지 → CrossRef → arXiv → 첫 페이지 레이아웃 휴리스틱 → 파일명 fallback의 6단계 파이프라인. `metadataSource`/`metadataConfidence`/`metadataWarnings`/`suggestedTags` 저장
- **저장/동기화**: Supabase Auth 사용자별 저장, localStorage 오프라인 폴백, dirty 기반 자동 저장, backoff 재시도
- **내보내기**: 포함 항목을 선택하는 Markdown/PDF 내보내기

### 3.2 읽기 구조화가 이미 구현된 영역

- **의미 라벨 하이라이트**: `주장 / 방법론 / 결과 / 한계·비판 / 질문·후속 확인 / 근거` 6종
- **인용 후보 보드**: 하이라이트에 인용 목적(`전제 인용 / 방법 참고 / 결과 비교 / 반론 / 한계 언급 / 관련 연구`)을 붙이면 보드에 자동 분류
- **수동 요약 템플릿**: 5문항 범용 템플릿(해결 문제/방법/결과/한계/핵심 이해) + 항목별 의미 라벨·인용 목적 연동
- **용어 사전**: 드래그 추가, 규칙 기반 용어 힌트, 선택형 AI 용어 설명(OpenRouter, 선택 기능)
- **진행 관리**: 5단계 리뷰 로드맵과 진행률

### 3.3 v3.1 로드맵에서 폐기/보류하는 항목

| v3.1 로드맵 항목 | v4.0 처분 |
| --- | --- |
| 요약 템플릿 AI 초안 | **폐기** (원칙 1 위배) |
| 논문 기반 Q&A | **폐기** (원칙 1 위배) |
| 다중 논문 비교·인용 네트워크 | **보류** (탐색 기능은 범위 밖. 단, "내 라이브러리 안" 비교는 §5.3에서 축소 재정의) |
| pdf.js 캔버스 직접 주석 | 유지 (읽기 경험 개선) |
| 공유·협업 | 보류 |

## 4. 원칙 1 — 요약 이외의 작업 자동화

"사용자가 읽고 생각하는 시간 외의 모든 시간을 줄인다." 요약을 AI에 맡기지 않는 대신, 요약 *주변부*의 반복 작업을 철저히 자동화한다.

### 4.1 자동화 대상 인벤토리

논문 1편 리뷰의 작업 흐름을 분해하면:

| 단계 | 작업 | 자동화 상태 | v4.0 계획 |
| --- | --- | --- | --- |
| 등록 | 파일 업로드/URL 등록 | ✅ 구현 | 유지 |
| 등록 | 메타데이터 입력 | ✅ 6단계 파이프라인 | 신뢰도 낮을 때 확인 UI 개선 |
| 등록 | 태그 부여 | 부분 (`suggestedTags`) | 제안 태그 원클릭 수락 UX |
| 추출 | 본문 텍스트 확보 | ✅ 추출+품질 진단 | 유지·운영 검증 지속 |
| 읽기 준비 | 섹션 구분 | 부분 (섹션 감지) | 섹션별 템플릿 연동 (§6) |
| 읽기 준비 | 봐야 할 지점 표시 | ❌ | **신규: 시그널 문장 스캐너 (§5.2)** |
| 정리 | 하이라이트 분류 | ✅ 의미 라벨 | 유지 |
| 정리 | 인용 후보 정리 | ✅ 목적 선택 시 자동 분류 | **신규: 라벨→목적 기본값 매핑 (§4.2)** |
| 정리 | 용어 정리 | ✅ 힌트+사전 | 유지 |
| 산출 | 노트 내보내기 | ✅ 항목 선택 내보내기 | **신규: 템플릿별 출력 형식 (§6.4)** |
| 산출 | 여러 논문 취합 | ❌ | **신규: 라이브러리 단위 취합 내보내기 (§5.3)** |
| 관리 | 진행률 추적 | ✅ 5단계 로드맵 | 템플릿별 완료 기준으로 재정의 (§6.3) |

### 4.2 신규 자동화: 라벨 → 인용 목적 기본값

하이라이트 의미 라벨과 인용 목적 사이에는 자연스러운 대응이 있다. 라벨을 붙이는 순간 인용 목적 기본값을 제안해 클릭 수를 줄인다.

| 의미 라벨 | 제안 기본 인용 목적 |
| --- | --- |
| 주장 | 전제 인용 |
| 방법론 | 방법 참고 |
| 결과 | 결과 비교 |
| 한계/비판 | 한계 언급 |
| 근거 | 전제 인용 |
| 질문/후속 확인 | (제안 없음 — 사용자 판단) |

제안은 언제나 덮어쓸 수 있는 기본값이며, 자동 확정하지 않는다.

### 4.3 자동화의 경계

- 자동화는 **규칙 기반·결정적(deterministic)** 을 우선한다. LLM 호출이 필요한 자동화는 "내용을 대신 이해하는가?"를 기준으로 심사하고, 그렇다면 도입하지 않는다.
- 자동 결과는 항상 근거(왜 이 태그/이 문장인가)와 함께 표시하고, 사용자 수정이 자동 결과보다 우선한다(`user_edited` 우선 원칙 유지).

## 5. 원칙 2 — 키워드·관점·한계·비판점 발굴 지원

논문 탐색을 도울 수 없는 대신, **확보한 논문 안에서 "무엇을 건져야 하는지"를 빨리 찾게** 한다. 사용자가 건져야 하는 것은 네 가지다.

- **키워드**: 이 논문의 축이 되는 개념·용어
- **관점**: 저자가 문제를 바라보는 프레임, 전제, 입장
- **한계**: 저자가 인정했거나 구조적으로 존재하는 제한점
- **비판점**: 독자로서 제기할 수 있는 반론·약점·미검증 지점

### 5.1 템플릿에 의한 발굴 (수동, 즉시 구현 가능)

§6의 목적별 템플릿 질문에 위 네 요소를 명시적으로 배치한다. 특히 "비판적 검토" 템플릿은 관점·한계·비판점 발굴 자체가 목적이다. 기존 하이라이트 라벨 6종 중 `한계/비판`, `질문/후속 확인`이 이미 이 흐름을 지원하므로, 템플릿 질문 ↔ 라벨 필터를 연동한다(예: "한계는?" 질문 카드에서 `한계/비판` 하이라이트만 모아 보기).

### 5.2 시그널 문장 스캐너 (규칙 기반 자동화, 신규)

본문에서 네 요소의 *후보 문장*을 규칙 기반으로 표시하는 기능. AI 요약이 아니라, 사용자가 봐야 할 위치를 하이라이트 후보로 안내하는 "형광펜 가이드"다.

- **키워드 후보**: 기존 용어 힌트(`HINT_PATTERN`) 확장 — 빈도·초록/키워드 섹션 등장 여부·대문자 약어를 조합해 상위 후보 목록 제시. 논문에 키워드 섹션이 있으면 그대로 파싱해 최우선 제시.
- **관점 시그널**: "we argue", "this paper proposes", "관점에서", "…를 전제로", "우리는 …라고 본다" 등 입장 표명 패턴이 있는 문장.
- **한계 시그널**: "limitation", "however", "future work", "제한점", "한계", "추후 연구", "…하지 못했다" 등 한계 표명 패턴이 있는 문장. Discussion/Conclusion 섹션 가중.
- **비판점 시그널**: 직접 탐지는 어렵다. 대신 (a) 표본/데이터 규모 언급 문장, (b) 가정 표명 문장("assume", "가정"), (c) 통계적 유의성 경계 문장을 "비판적으로 볼 후보"로 표시하고, 비판 자체는 사용자가 `한계/비판` 라벨·템플릿으로 작성한다.

동작 원칙:

- 스캐너 결과는 원문 패널에서 점선 밑줄 등 **비침습적 표시**로 보여주고, 클릭 한 번으로 해당 라벨의 하이라이트로 승격한다.
- 결과는 저장하지 않는 휘발성 안내다(사용자가 승격한 하이라이트만 저장). 오탐이 있어도 사용자 노트를 오염시키지 않는다.
- 서버 추출 파이프라인이 아니라 프론트엔드 규칙 모듈(`frontend/src/lib/`)로 구현해 테스트 가능하게 유지한다.

### 5.3 라이브러리 단위 취합 (탐색이 아닌 "회수")

논문을 찾아 주지는 못해도, **이미 읽은 논문에서 무엇을 건졌는지**는 한눈에 회수할 수 있어야 한다.

- 내 라이브러리 전체에서 라벨/인용 목적별 취합 뷰: 예) "모든 논문의 `한계/비판` 하이라이트", "모든 `반론` 인용 후보"
- 취합 결과의 Markdown 내보내기(논문별 출처 표기 포함) — 선행연구 정리 장(章) 초안의 재료가 된다.
- 이는 v3.1의 "다중 논문 비교·인용 네트워크"를 탐색 기능 없이 성립하는 최소 형태로 축소 재정의한 것이다.

## 6. 원칙 3 — 작업 목표별 목적 특화 템플릿

### 6.1 설계 방침

- 템플릿 하나가 모든 상황을 덮으려 하지 않는다. **템플릿 = (목적, 질문 세트, 권장 라벨, 완료 기준, 출력 형식)** 의 묶음으로 정의한다.
- 노트 생성 시(또는 이후 변경 가능) 사용자가 "이 논문을 왜 읽는가"를 선택하면 해당 템플릿이 적용된다.
- 기존 범용 5문항 템플릿은 "빠른 훑어보기" 템플릿으로 남겨 하위 호환을 유지한다.

### 6.2 초기 템플릿 세트 (5종)

**T1. 빠른 훑어보기 (기존 범용 템플릿 계승)**
- 목적: 30분 내 논문의 뼈대 파악, 정독 여부 판단
- 질문: 무엇을 해결하나 / 방법은 / 결과는 / 한계는 / 내가 이해한 핵심은 (기존 5문항 유지)
- 권장 라벨: 주장, 결과
- 완료 기준: 5문항 작성

**T2. 선행연구 정리 (Related Work 집필 준비)**
- 목적: 내 논문의 서론/선행연구 절에 인용할 재료 수집
- 질문: 이 논문이 내 연구와 어떻게 연결되나 / 내 논문에서 이 논문을 한 문장으로 소개한다면 / 내 연구와의 차별점은 / 어떤 맥락(전제·비교·반론)에서 인용할 것인가
- 권장 라벨: 주장, 근거 / 권장 인용 목적: 전제 인용, 관련 연구
- 완료 기준: 인용 후보 1개 이상 + "한 문장 소개" 작성
- 출력: 인용 후보 보드 중심 Markdown

**T3. 방법론 벤치마킹**
- 목적: 이 논문의 연구 설계·실험·분석 방법을 내 연구에 이식/비교
- 질문: 연구 설계는(대상·표본·조건) / 핵심 절차·도구·지표는 / 내 상황에 적용 시 바꿔야 할 것은 / 이 방법의 전제 조건과 취약점은
- 권장 라벨: 방법론 / 권장 인용 목적: 방법 참고
- 완료 기준: 방법론 하이라이트 3개 이상 + 적용 계획 질문 작성

**T4. 비판적 검토 (관점·한계·비판점 발굴 — §5와 직결)**
- 목적: 리뷰·세미나 발제·반론 구성. 논문을 뜯어보는 읽기
- 질문: 저자의 관점/전제는 무엇인가 / 저자가 인정한 한계는 / 저자가 말하지 않은 한계는 / 주장-근거 사슬에서 가장 약한 고리는 / 결과가 성립하지 않는 조건은
- 권장 라벨: 한계/비판, 질문/후속 확인, 근거 / 권장 인용 목적: 반론, 한계 언급
- 완료 기준: `한계/비판` 하이라이트 2개 이상 + "말하지 않은 한계" 작성
- 시그널 문장 스캐너(§5.2)를 기본 활성화하는 유일한 템플릿

**T5. 결과 비교·수치 수집**
- 목적: 성능표·효과크기 등 내 결과와 비교할 수치 수집
- 질문: 비교 대상 지표와 값은 / 실험 조건이 내 것과 같은가 다른가 / 직접 비교 가능한가, 보정이 필요한가
- 권장 라벨: 결과 / 권장 인용 목적: 결과 비교
- 완료 기준: `결과 비교` 인용 후보 1개 이상

### 6.3 진행 로드맵과의 연동

5단계 리뷰 로드맵의 완료 판정을 템플릿별 완료 기준으로 계산한다. 예: T4에서는 `한계/비판` 하이라이트 수가 진행률에 반영되고, T1에서는 5문항 작성률이 반영된다. (`frontend/src/lib/reviewProgress.ts` 확장)

### 6.4 출력 형식 연동

내보내기(Markdown/PDF)의 기본 포함 항목·순서를 템플릿이 결정한다. 예: T2는 인용 후보 보드가 맨 위, T3는 방법론 하이라이트와 적용 계획이 맨 위. 기존 포함 항목 체크박스는 수동 오버라이드로 유지한다.

### 6.5 데이터 모델 영향

- 노트에 `templateId`(기본값 `t1_quick`)와 템플릿별 답변 저장 구조 추가. 기존 노트는 T1으로 마이그레이션(질문 키가 동일하므로 무손실).
- 템플릿 정의는 `frontend/src/constants.ts`(또는 분리 모듈 `frontend/src/lib/templates.ts`)에 선언적 데이터로 두고, 백엔드는 스키마 변경 없이 노트 JSON(`review_notes`) 안에 저장한다 — 서버 마이그레이션 불필요.

## 8. 성공 지표

- 논문 1편 등록→읽기 시작까지 사용자 개입 횟수 (자동화 지표, 감소 목표)
- 템플릿 선택률: T1 이외 목적 템플릿을 선택한 노트 비율 (목적 특화가 실제로 쓰이는가)
- T4 노트에서 `한계/비판`·`질문/후속 확인` 하이라이트 수 (비판점 발굴 지표)
- 시그널 스캐너 후보 → 하이라이트 승격률 (스캐너 정밀도 대리 지표; 낮으면 규칙 축소)
- 내보내기 실행률 (노트가 실제 산출물로 이어지는가)

## 9. 유지되는 운영 원칙

- 사용자 데이터 보존 최우선: 자동화가 사용자 입력을 덮어쓰지 않는다 (`user_edited` 우선).
- 추출 실패 시에도 등록을 막지 않고 직접 입력 경로를 연다.
- Supabase Auth + 사용자별 저장, localStorage 폴백, dirty 기반 자동 저장 등 v3.1까지 구축한 저장/인증 견고화는 그대로 유지한다.
- AI 사용은 용어 설명 초안(선택 기능, `AI_API_KEY` 설정 시)에 한정하며, 신규 AI 기능 도입 시 "내용을 대신 이해하는가" 심사를 거친다.
