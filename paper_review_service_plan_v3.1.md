# 논문 리뷰·요약 웹 서비스 기획서

> 작성일: 2026-06-14 | 버전: v3.1 | 작성자: PM

> **v3.0 변경 요지:** AI의 역할을 "자동 생성자"에서 "요청 시 돕는 보조"로 격하. 사용자의 직접 작성을 제품의 중심에 둔다. **핵심 전략적 이유: AI API 도입이 지연될 수 있으므로, AI 없이도 가치를 주는 제품으로 설계하고 AI는 선택적 레이어로 분리한다.** 이로써 API 일정과 무관하게 MVP를 출시할 수 있다.

> **v3.1 수정 요지:** 레이어 분류 오류(인용 네트워크) 정정, 인증 스택을 Vite 호환으로 교체, 사용자 질문 기록을 코어 기능으로 분리, 스토리보드 화면 번호 재정렬, 용어 힌트의 한계 명시.

## 1. 서비스 개요

**서비스명(가칭):** PaperLens

**한 줄 정의:** 사용자가 직접 작성하는 논문 리뷰 노트 도구. AI는 요청할 때만 거드는 보조.

**v3.0 핵심 컨셉:** 논문을 읽고 내 언어로 정리하는 과정의 주인공은 사용자다. PaperLens는 그 작성 과정을 매끄럽게 만드는 워크스페이스이며, AI는 사용자가 원할 때 초안·설명을 제안하는 보조 도구일 뿐이다. **AI가 빠져도 제품은 동작한다.**

## 3. 최근 트렌드 분석

### 3-1. 시장 트렌드
- AI 연구 보조 도구가 법률·교육·의료 등 전문 분야로 급속 확장 중
- 2026년 AI 핵심 기조: "화려함(Flair) → 기능(Function)" - 실질적 ROI 중심으로 전환
- "AI 자동 생성물에 대한 신뢰 피로(trust fatigue)" 부상 → 사용자가 직접 검증·작성하는 도구 선호 증가
- 능동적 학습(active recall) 도구가 수동적 요약 소비보다 학습 효과 측면에서 재평가받는 흐름

## 4. 기획 방향 (To-Be)

> **차별점: 한국어 최적화 + 사용자 주도 리뷰 워크플로우 + AI 비의존 설계**

### 4-1. AI 보조 3원칙 (v3.0 핵심)

1. **사용자가 주체** — 사용자가 먼저 읽고 쓰고, AI는 그 다음에 거든다
2. **요청 시에만 작동** — AI는 자동 실행되지 않는다. 사용자가 버튼을 눌러야 작동
3. **수락을 거쳐 확정** — 모든 AI 출력은 사용자의 "수락" 행동을 거쳐야 노트에 들어간다. AI 생성물이 그대로 결과물이 되지 않는다

### 4-2. AI 비의존 설계 (전략적 핵심)

AI API 도입이 지연될 수 있다는 리스크에 대응해, 제품을 두 레이어로 분리한다.

| 레이어 | 내용 | AI 의존 |
|--------|------|---------|
| **코어 레이어** | 논문 등록, 수동 작성 도구, 하이라이트, 메모, 저장 | ❌ 없음 (AI 없이 동작) |
| **보조 레이어** | AI 초안, 용어 설명, Q&A | ✅ 선택적 (나중에 얹기 가능) |

> 코어 레이어만으로도 "논문 정리 워크스페이스"로서 가치가 있다. AI 보조 레이어는 API 준비가 되는 대로 점진 도입한다.

### 4-3. 핵심 기능 및 구현 가능성

| 기능 | 설명 | AI 의존 | 구현 가능성 |
|------|------|---------|-------------|
| PDF 업로드 & 파싱 | PDF → 텍스트 추출 | ❌ | ✅ 높음 (PyMuPDF, pdf.js) |
| 핵심 문장 하이라이트 | 사용자가 본문 드래그하여 지정 | ❌ | ✅ 높음 (pdf.js) |

| 핵심 용어 사전 | 사용자가 모르는 단어 지목, AI 설명은 보조 | 선택 | ✅ 높음 |
| 수동 요약 템플릿 | 5가지 질문 사용자 작성 | ❌ | ✅ 높음 (UI 중심) |

## 5. Pain Point

1. **시간 비용** - 논문 1편 읽고 이해하는 데 평균 2~4시간 소요
2. **영어 장벽** - 비영어권 대학원생의 전문 영어 독해 부담
3. **맥락 파악 어려움** - 논문 하나만으로는 연구 흐름 파악 한계
4. **리뷰 작성 막막함** - 어떻게 비판·요약해야 할지 모르는 초보 연구자
5. **정보 파편화** - 읽은 논문, 메모, 인용이 여러 도구에 분산됨
6. **재현 불가** - "저번에 읽었는데 어디 저장했더라?" 문제
7. **AI 요약 불신** - AI 요약을 그대로 믿기 어렵고 검증 과정이 분리되어 비효율

> **v3.0 통찰:** Pain #4·#7은 AI 자동화로 풀리지 않는다. 오히려 **사용자가 직접 쓰되 막힐 때만 AI가 거드는 구조**가 답이다. 이는 학습 효과(능동 회상)와 신뢰 문제를 동시에 해결한다.

> **Pain Point별 해결 단계:** 코어 MVP(Phase 1)는 작성·정리에서 오는 불편(#4 리뷰 작성 막막함, #5 정보 파편화, #6 재현 불가)을 우선 해결한다. 독해 부담(#1 시간 비용, #2 영어 장벽)은 AI 보조(Phase 2)가 본격적으로 다룬다. 즉 코어만으로도 "정리 도구"로서 가치가 서고, AI는 "독해 가속기"로 가치를 더한다.

## 7. 페르소나

### 페르소나 A — "논문 폭탄 맞은 석사생" 김지수 (26세)

- **직업/상황:** 생명과학 석사 1년차, 지도교수에게 주 3편 논문 리뷰 과제
- **목표:** 내가 이해한 내용으로 리뷰 노트를 직접 작성하되, 막히는 부분만 도움받기
- **기술 수준:** ChatGPT는 써봤지만 논문 분석에 어떻게 쓰는지 모름
- **Pain Point:**
  - 영어 논문 독해에 시간 대부분 소모
  - AI 요약을 그대로 제출하기엔 불안하고, 내 언어로 정리하고 싶음
  - 모르는 전문 용어가 너무 많음
- **기대 가치:** "내가 직접 쓰면서, 모르는 단어만 클릭해서 설명 받으면 학습도 되고 안심도 된다"

### 페르소나 B — "빠른 시장조사 필요한 스타트업 PM" 이준혁 (34세)

- **직업/상황:** AI 헬스케어 스타트업 PM, 투자자 미팅 전 최신 논문 트렌드 파악 필요
- **목표:** 핵심 문장만 빠르게 추리고 내 인사이트를 메모로 기록
- **기술 수준:** SaaS 툴 적극 활용, 유료 결제 의향 있음
- **Pain Point:**
  - 논문 읽을 시간 없음, 핵심만 빠르게 필요
  - 영어 논문을 팀원에게 공유할 한국어 메모 필요
- **기대 가치:** "본문에서 중요한 문장만 하이라이트하고, 섹션별로 짧게 메모"

## 8. 리뷰 노트 화면 구조

> 논문 1편당 하나의 **리뷰 노트**가 생성된다. 9개 영역으로 구성되며, 대부분은 사용자 작성 영역이고 AI는 일부 영역에서 선택적으로 거든다.

### 8-1. 리뷰 노트 9개 구성 영역

| # | 영역 | 주 작성 주체 | AI 역할 |
|---|------|--------------|---------|
| 1 | 논문 메타정보 | 자동 추출 | - |
| 2 | 한 줄 요약 | 사용자 작성 | 요청 시 초안 제안 (보조) |
| 3 | 섹션별 요약 | 사용자 작성 | 요청 시 초안 제안 (보조) |
| 4 | 핵심 용어 사전 | **사용자가 모르는 단어 지목** | 지목된 단어만 설명 (보조) |
| 5 | 읽으며 생긴 질문 | 사용자 작성 | 요청 시 답변 (보조) |
| 6 | 핵심 문장 하이라이트 | **사용자 직접 드래그** | - (코어 기능) |
| 7 | 수동 요약 템플릿 | 사용자 작성 | 요청 시 초안 제안 (보조) |
| 8 | 섹션별 메모 카드 | 사용자 작성 | - (코어 기능) |
| 9 | 전체 리뷰 노트 | 통합 결과 | - |

### 8-2. 핵심 용어 사전 — 사용자 주도 재설계 (v3.0 핵심 변경)

**기존: AI가 어려운 용어를 골라줌 → 변경: 사용자가 "내가 모르는 단어"를 직접 지목**

| 구분 | 이전 (AI 주도) | v3.0 (사용자 주도) |
|------|----------------|---------------------|
| 시작점 | AI가 용어 후보 제시 | 사용자가 본문에서 모르는 단어 드래그 |
| AI 역할 | 용어 추출 + 설명 | 지목된 단어만 설명 (요청 시) |
| 노트 등재 | 사용자가 후보 중 선택 | 사용자가 추가한 단어만 등재 |

**동작 흐름:**
1. 사용자가 본문을 읽다가 모르는 단어를 드래그
2. "📖 용어 사전에 추가" 버튼 노출
3. 추가 시 AI가 그 단어에 한해 쉬운 설명 생성 (보조, 선택)
4. 사용자가 설명 확인·편집 → 노트에 확정

**보완 (초보자 길잡이):** 본문에서 전문 용어로 추정되는 단어에 **옅은 밑줄 힌트**만 표시한다. 강요하지 않으며, 사용자는 클릭하든 무시하든 자유. "내가 뭘 모르는지도 모르는" 초보자가 길을 잃지 않게 하는 최소한의 안내.

> **힌트 정확도의 한계:** 초기 힌트는 규칙 기반(영문 약어·외래어 패턴)이라 정확도가 제한적이다. 특히 한국어 순한글 전문용어(예: "되먹임", "들뜸")는 규칙으로 잡기 어렵다. 힌트는 어디까지나 보조 안내이며, 모르는 단어를 최종 지목하는 주체는 사용자다. 정확도 개선은 Phase 2 이후 용어 사전 누적 데이터·AI 보조로 점진 보강한다.

**부가 가치:** 사용자별 "모르는 단어" 데이터가 쌓이면, 향후 개인 맞춤 난이도 조정·반복 학습 기능의 기반 데이터가 된다.

### 8-3. 수동 요약 템플릿 (영역 7 상세)

사용자가 직접 답변하는 5가지 질문. 본인의 이해를 기록하는 것이 목적.

1. 이 논문은 무엇을 해결하려 하는가?
2. 어떤 방법을 사용했는가?
3. 결과는 무엇인가?
4. 한계는 무엇인가?
5. 내가 이해한 핵심은 무엇인가?

> 각 질문 옆 "✨ AI 초안" 버튼은 보조. 기본은 빈 칸에 직접 작성하는 것이며, 막힐 때만 초안을 호출한다.

### 8-4. 섹션별 메모 카드 (영역 8 상세)

논문 전체를 한 번에 요약하는 대신, 구역별로 짧은 메모를 남긴다. 순수 사용자 작성 영역(AI 미개입).

| 구역 | 메모 카드 용도 |
|------|----------------|
| Abstract | 전체 인상, 읽을 가치 판단 |
| Introduction | 연구 동기, 기존 연구와의 차별점 |
| Method | 방법론에 대한 의문·평가 |
| Result | 결과 해석, 인상적인 수치 |
| Discussion | 저자 주장 동의·반론, 응용 아이디어 |

## 11. 기능 명세 (Functional Specification)

### FS-01: 논문 등록 및 파싱 [코어]

- **입력:** PDF 파일(최대 50MB, 200페이지) 또는 DOI/URL
- **처리:** 유효성 검사 → 텍스트 추출(PyMuPDF) → 메타정보 추출(DOI 시 CrossRef API) → 섹션 자동 분류 → DB 저장 + 빈 리뷰 노트 생성
- **출력:** 본문 뷰어 + 빈 리뷰 노트
- **예외:** 스캔 PDF → OCR 안내 / 암호화 PDF → 오류

### FS-02: 핵심 문장 하이라이트 [코어]

- **입력:** 사용자의 본문 드래그
- **처리:** 드래그 영역 → "하이라이트 추가" 버튼 → 리뷰 노트 영역 6에 누적 저장
- **출력:** 하이라이트 문장 목록 (출처 위치 포함)

### FS-03: 핵심 용어 사전 [코어 + 보조]

- **입력:** 사용자가 드래그한 모르는 단어
- **처리:**
  1. [코어] 단어 드래그 → "용어 사전에 추가" → 빈 설명 칸과 함께 등재
  2. [코어] 전문 용어 추정 단어에 옅은 밑줄 힌트 표시 (규칙 기반: 대문자 약어, 외래어 등)
  3. [보조] "✨ 설명 받기" 클릭 시 AI가 해당 단어 설명 생성
  4. 사용자가 설명 확인·편집 → 확정
- **출력:** 용어-설명 쌍 목록 (사용자 확정본)

### FS-04: 수동 작성 영역 (요약/템플릿/메모) [코어]

- **입력:** 사용자 텍스트 입력
- **처리:** 자동 저장(5초 debounce) / 빈 칸일 때만 "✨ AI 초안" 버튼 노출 [보조]
- **출력:** 사용자 작성 콘텐츠 (AI 초안 사용 시 `초안`→`내가 작성` 뱃지 전환)

### FS-05: AI 보조 레이어 [보조, 선택적]

- **전제:** AI API 연동 완료 시에만 활성화. 미연동 시 버튼 비활성("준비 중")
- **처리:** 사용자 요청(버튼 클릭) → LLM 호출 → 결과를 편집 가능 상태로 제시 → 사용자 수락 시 노트 확정
- **원칙:** 자동 실행 금지, 수락 전 노트 미반영

## 13. ERD (Entity Relationship Diagram)

> v3.0: REVIEW_NOTE 중심. TERM에 사용자 주도 필드(added_by_user) 추가.

**관계 구조**

| 관계 | 설명 |
|------|------|
| USER 1:N PAPER | 사용자는 여러 논문 등록 |
| PAPER 1:1 REVIEW_NOTE | 논문 1편당 리뷰 노트 1개 |
| REVIEW_NOTE 1:N | SECTION_SUMMARY / HIGHLIGHT / TERM / QUESTION / MEMO_CARD |
| REVIEW_NOTE 1:1 | MANUAL_TEMPLATE (5문항 양식) |

**엔티티별 주요 컬럼**

| 엔티티 | 주요 컬럼 |
|--------|-----------|
| PAPER | id(PK), user_id(FK), title, authors, source_url(DOI), file_url, text_extracted, sections(JSON), upload_at |
| REVIEW_NOTE | id(PK), paper_id(FK), user_id(FK), status(draft/completed), created_at, updated_at |
| HIGHLIGHT | id(PK), note_id(FK), sentence_text, source_position, created_at |
| TERM | id(PK), note_id(FK), term, explanation, added_by_user(bool), ai_explained(bool), created_at |
| SECTION_SUMMARY | id(PK), note_id(FK), section_type, content, source(user/ai_draft), created_at |
| MEMO_CARD | id(PK), note_id(FK), section_type(Abstract/Intro/Method/Result/Discussion), memo_content, created_at |
| QUESTION | id(PK), note_id(FK), question, ai_answer, sources(JSON), created_at |
| MANUAL_TEMPLATE | id(PK), note_id(FK), q1_problem, q2_method, q3_result, q4_limitation, q5_my_takeaway, created_at |

> TERM.added_by_user = 사용자가 직접 추가했는지 여부. TERM.ai_explained = AI 설명을 받았는지 여부. 두 필드로 "사용자가 모르는 단어" 데이터를 추적한다.

## 16. 개발 우선순위 (Roadmap)

> v3.0: AI 비의존 코어를 먼저 완성하고, AI 보조를 나중에 얹는다. API 일정과 출시 일정을 분리.

| 단계 | 기간 | 내용 | AI 의존 |
|------|------|------|---------|
| **Phase 1 — 코어 MVP** | 1~2개월 | 논문 등록 → 하이라이트 → 용어 사전(수동 추가+힌트) → 수동 작성 영역 → 저장 | ❌ 없음 |
| **Phase 2 — AI 보조 도입** | 3~4개월 | 용어 AI 설명, 요약 AI 초안, 템플릿 AI 초안, 논문 Q&A | ✅ 선택 |
| **Phase 3 — 확장** | 5~6개월 | 다중 논문 비교, 인용 네트워크, 팀 협업·공유, 개인 맞춤 학습 | 일부 |

### Phase 1 코어 MVP 우선순위 상세 (검증 속도 기준)

1. **PDF 업로드 & 파싱** — 진입점, 난이도 낮음
2. **핵심 문장 하이라이트** — pdf.js 드래그, AI 불필요한 즉시 가치
3. **핵심 용어 사전(수동 추가 + 밑줄 힌트)** — 사용자 주도 차별화 핵심
4. **수동 작성 영역(요약/템플릿/메모)** — 작성 UX 마찰 최소화가 관건

> **전략 포인트:** Phase 1은 AI API 없이 완성·출시 가능하다. AI API 도입이 늦어져도 코어 MVP로 사용자 검증을 먼저 시작할 수 있다.
